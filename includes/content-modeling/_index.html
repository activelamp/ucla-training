Day 1 Review

Structured Content

  • Do not think of your site in terms of pages, think in terms of content.
  • Content can take many shapes... (Home Page, Sidebar, RSS, etc...)
  • Talk about the importance of having a content model
  • Form follows function, you can build functionality first
  • Editors can begin getting used to entering content before frontend is done

COPE

Create Once, Publish Everywhere

https://www.programmableweb.com/news/cope-create-once-publish-everywhere/2009/10/13
  • Holy grail
  • Not really pragmatic, but something to strive for
  • Think of Drupal as a CMS and a Web Publishing Tool

Keep content seperate from display

Content should be agnostic of display

  • Webpage
  • Mobile App
  • Voice App (Alexa, Siri, etc...)

For the geeks

This sounds like MVC (Model View Controller pattern)

  • The model data can be represented by many different views
  • The model can be updated more than one way
  • Think of your content like an API

  • Program
    • Does the Program really have an address, or is that address for the Campus unit? another content type
    • Will the site every want to show more information for Campus Unit? another content type
    • Point out that the address fields are really one field (collection)
    • Could the same Manager be on more than one program? another content type.
    • Could the same Follow-up be on more than one program? another content type.

    • Why are the description fields the same as on program? can it be another aggregation?
    • Project does not need the Category taxonomy, it is inherited from Program
  • Partner
    • I think we should create a vocabulary heirarchical Legislative Districts vocabulary.
    • Partner Type does not need to be a Taxonomy since it only ever applies to partners, just make it a field.
    • Writing a service to automatically assign the legislative district would be a good custom module to write when saving the partner.
